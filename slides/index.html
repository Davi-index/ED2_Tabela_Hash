Sistema de Consulta de Usuários em Larga Escala

Como encontrar uma pessoa entre milhões em frações de segundo

Um programa em C que combina duas técnicas para responder, de forma rápida e confiável, à pergunta: "o usuário X existe e quais são seus dados?"

Autor(es):Davi Alves Maia
Disciplina: Algoritmos e Estruturas de Dados II
Data: 30/07/2026

O Problema

Precisamos ser rápidos — e muito

Imagine um sistema com milhões de usuários cadastrados. A cada segundo chegam centenas de perguntas como:

"O usuário de ID 4.872.301 existe? Quais são os seus dados?"

Verificar um por um, da primeira à última linha, não funciona em escala. Uma busca linear demoraria tempo proporcional ao tamanho da lista — inviável.

Problema 1
Encontrar um ID que existe rapidamente.
Problema 2
Descartar um ID que não existe sem gastar tempo procurando.
A Solução

Duas estruturas, dois papéis

🔦

Filtro de Bloom

Um guarda da entrada. Responde em frações de segundo: "esse ID definitivamente não existe — nem precisa procurar" ou "pode existir — pode entrar e verificar".

📇

Tabela Hash

Um fichário organizado. Quando o Bloom libera a entrada, a tabela hash vai direto ao local exato do registro, sem percorrer todos os outros.

Juntas, evitam trabalho desnecessário antes de qualquer busca e garantem a resposta exata quando necessário.
Entendendo o Filtro de Bloom

O guarda da catraca

Pense num show com lista VIP. O segurança na catraca tem um livro de anotações rápidas (muito menor que a lista completa). Ele consulta esse livro em segundos:

✗
"Esse nome certamente não está na lista." — O segurança manda embora sem nem chamar o gerente. Rápido, definitivo.
?
"Pode estar na lista — vou confirmar com o gerente." — Aí sim consulta a lista completa (a tabela hash).
Detalhe importante: o livro de anotações pode se enganar para um lado — pode dizer "pode estar" quando na verdade não está (falso positivo). Mas nunca nega alguém que realmente está na lista. O gerente (a hash) sempre dá a palavra final.
Entendendo a Tabela Hash

O fichário com gavetas numeradas

Imagine uma biblioteca com gavetas numeradas. Cada ficha vai para uma gaveta calculada a partir do número do livro — sempre a mesma gaveta para o mesmo número:

1
O sistema pega o ID do usuário e aplica um cálculo matemático simples.
2
O resultado indica qual gaveta verificar — direto, sem percorrer todas.
3
Se dois IDs caírem na mesma gaveta (colisão), os registros ficam juntos numa pequena lista dentro da gaveta.
Com as gavetas bem distribuídas, cada uma tem no máximo 1 ou 2 fichas. A busca é praticamente instantânea — tempo constante, independente do tamanho total.
Como Funciona na Prática

O caminho de uma consulta

Você digita: consultar ID 4.872.301 │ ▼ ┌─────────────────────────────────┐ │ FILTRO DE BLOOM │ │ "Já vi esse ID antes?" │ └─────────────────────────────────┘ │ │ CERTAMENTE NÃO PROVAVELMENTE SIM │ │ ▼ ▼ DESCARTADO ┌──────────────────────┐ Resposta: │ TABELA HASH │ "Não existe" │ Vai direto à gaveta │ (< 0,1 µs) └──────────────────────┘ │ │ ENCONTRADO NÃO ENCONTRADO Retorna Bloom errou os dados (falso positivo, muito raro)
Os Dados do Usuário

O que é guardado por registro

Cada usuário cadastrado possui cinco campos:

  • ID — número único de identificação (chave de busca)
  • Nome — até 128 caracteres
  • E-mail — até 256 caracteres
  • Idade — número inteiro
  • CPF — até 20 caracteres

Exemplo de arquivo de entrada

# linhas com # são comentários
1;Ana Souza;ana@email.com;28;123.456.789-00
2;Carlos Lima;carlos@email.com;34;987.654.321-00
7;Beatriz Nunes;bea@email.com;22;111.222.333-44

Campos separados por ponto e vírgula, um usuário por linha.

Manutenção Automática

Quando as gavetas ficam cheias

Se muitos registros acabam na mesma gaveta, a busca fica mais lenta. O programa monitora isso automaticamente pelo fator de carga:

Fator de carga = quantidade de usuários ÷ número de gavetas
!
Quando o fator ultrapassa 0,75 (gavetas ficando cheias), o sistema dobra o número de gavetas automaticamente e redistribui todos os registros.
✓
Resultado: as gavetas ficam sempre com poucos registros, mantendo a busca rápida mesmo com milhões de usuários.
Esse processo se chama rehash. O custo é baixo porque o programa apenas reorganiza os ponteiros — não recria os registros do zero.
Comparação de Desempenho

Por que essa combinação faz diferença

Situação Busca simples (lista) Hash + Bloom
ID existe (1 M registros) Até 500.000 comparações 1–2 comparações
ID não existe 1.000.000 comparações (percorre tudo) Descartado pelo Bloom em < 0,1 µs
Inserção de novo usuário Inserção ao final Direto na gaveta correta
Em sistemas reais, a maioria das consultas é por IDs que não existem (tentativas inválidas, erros de digitação). O Bloom descarta esses casos antes mesmo de tocar nos dados.

Como Usar o Programa

Simples de compilar e rodar

O projeto foi desenvolvido em linguagem C, sem dependências externas. Para usá-lo, basta ter um compilador C (GCC) instalado:

make               # compila o programa
make run           # compila e já abre com o arquivo de exemplo
./bin/consulta arquivo.txt # carrega qualquer arquivo de usuários
Zero erros e zero avisos na compilação, mesmo com as checagens mais rígidas ativadas.
Funciona em qualquer sistema Linux/macOS com GCC. Sem bibliotecas externas para instalar.

Honestidade Técnica

O que o sistema ainda não faz

Limitações atuais

  • Os dados ficam apenas na memória — ao fechar o programa, o que foi adicionado manualmente é perdido.
  • O filtro de Bloom não suporta remoção de usuários (marcações de bit não podem ser "desfeitas" facilmente).
  • A busca é somente por ID — não é possível buscar por nome, CPF ou e-mail.

Próximos passos possíveis

  • Salvar e carregar os dados em formato binário (persistência).
  • Adicionar busca secundária por e-mail ou CPF.
  • Gerar um arquivo com 1 milhão de registros para demonstrar a escala real.
  • Versão com múltiplas consultas simultâneas (threads).
Conclusão

O que foi alcançado

  • Um programa funcional em C que responde consultas em menos de 1 microssegundo em média.
  • O filtro de Bloom elimina IDs inexistentes antes de qualquer busca, poupando processamento.
  • A tabela hash garante acesso direto e exato ao registro quando necessário.
  • Código modular, comentado e sem dependências externas — fácil de entender e expandir.
  • As estatísticas em tempo real permitem observar a teoria funcionando em números reais.
"Não otimize a busca — evite buscar quando não for preciso."

Anexo — Perguntas Prováveis

Respostas rápidas para a banca

O Bloom pode dar uma resposta errada?
Só num sentido: pode dizer "provavelmente existe" quando não existe (falso positivo, < 1% dos casos). Nunca diz "não existe" para algo que foi cadastrado. A hash sempre confirma.
Por que não usar somente a tabela hash?
A hash funciona muito bem para IDs que existem. O Bloom poupa o acesso à hash quando o ID não existe — evita percorrer a lista de uma gaveta sem necessidade.
O que acontece quando a tabela cresce demais?
O rehash dobra o número de gavetas automaticamente quando necessário, mantendo a busca sempre rápida. O custo é diluído ao longo das inserções.
Por que C e não Python ou Java?
C permite controle direto de memória e aritmética de bits, essencial para implementar o vetor de bits do Bloom de forma eficiente e medir tempos reais em microssegundos.
